# MindX Challenge — AI-Driven Project Git Commit Strategy & Rules

## Purpose

This document defines the Git workflow, commit strategy, branch naming convention, and pull request standards for the project.

The goal is to ensure:

- Clean and professional Git history
- Easy review process for MindX reviewers
- Clear development progress tracking
- Production-level engineering practices
- AI coding agents can follow a consistent workflow

---

# AI Project Context

This project is being developed with assistance from AI coding agents.

The repository must clearly demonstrate:

- Human architectural thinking
- Proper task decomposition
- AI-assisted engineering workflow
- Production-level backend practices
- Clean collaboration between human and AI
- Ability to control AI-generated code quality

MindX reviewers should be able to understand:

- What was designed manually
- What was generated by AI agents
- How the development process was controlled
- How engineering decisions evolved over time

# Core Principles

## 1. Every Commit Must Have Purpose

Each commit should represent:

- One feature
- One fix
- One refactor
- One infrastructure change
- One logical development step

Avoid mixing unrelated changes in a single commit.

---

## 2. Small and Reviewable Commits

Good commits:

- Easy to review
- Easy to rollback
- Easy to debug
- Easy to understand from commit history

Preferred commit size:

- 30 minutes → 2 hours of work
- One completed flow
- One completed endpoint
- One infrastructure setup

---

## 3. Commit by Business Feature

DO NOT commit by technical layer.

Bad:

```bash
feat(api)
feat(database)
feat(ui)
```

Good:

```bash
feat(cart): implement add to cart flow
```

The commit should include:

- route
- controller
- service
- validation
- schema update
- UI integration (if needed)

---

# Git Branch Strategy

## Main Branches

```bash
main
develop
```

---

## Feature Branches

All development must happen in feature branches.

Format:

```bash
feature/<feature-name>
```

Examples:

```bash
feature/auth
feature/product-catalog
feature/cart
feature/payment
feature/admin-dashboard
feature/order-flow
```

---

## Fix Branches

Format:

```bash
fix/<issue-name>
```

Examples:

```bash
fix/payment-webhook
fix/duplicate-order
fix/cart-race-condition
```

---

## Refactor Branches

Format:

```bash
refactor/<module-name>
```

Examples:

```bash
refactor/order-service
refactor/auth-module
```

---

## Format

# AI Agent Development Workflow

## AI Usage Philosophy

AI agents are development accelerators.

However:

- Architecture decisions must remain consistent
- Git history must remain human-readable
- Features must remain logically grouped
- AI-generated code must still follow engineering standards

The repository should NOT look like random AI-generated output.

It should look like a professionally managed engineering project assisted by AI.

---

# AI Agent Commit Rules

When AI agents generate code for this project:

## MUST

- Review AI-generated code before commit
- Ensure generated files follow project architecture
- Keep service boundaries clean
- Keep business logic separated from controllers
- Ensure AI-generated code follows naming conventions
- Commit by feature, not by AI generation session

- Commit only after a complete logical unit
- Use conventional commits
- Keep commits isolated
- Avoid unrelated file changes
- Group files by feature
- Maintain clean Git history

---

## MUST NOT

- Commit generated temporary files
- Commit debugging logs
- Commit commented-out code
- Commit unfinished partial flows
- Commit unrelated formatting changes

# AI Project Specific Rules

## 1. Never Commit Massive AI Dumps

BAD:

```bash
feat: generate ecommerce backend
```

This instantly looks AI-generated and unmanaged.

---

## 2. Split AI Output Into Logical Engineering Steps

GOOD:

```bash
feat(auth): implement JWT authentication
feat(product): implement product catalog api
feat(cart): implement cart persistence
feat(order): implement checkout transaction flow
```

This makes the repository look engineered instead of auto-generated.

---

## 3. Refactor Generated Code Frequently

AI-generated code usually:

- duplicates logic
- mixes responsibilities
- creates large files
- violates clean architecture

Therefore refactor commits are extremely important.

Examples:

```bash
refactor(payment): separate webhook verification service
refactor(auth): extract token generation utility
refactor(order): move business rules into domain layer
```

---

## 4. Use Commits to Show Engineering Maturity

For AI-assisted projects, reviewer confidence increases when they see:

```bash
feat(redis): add distributed locking
feat(payment): implement idempotency handling
fix(order): prevent duplicate checkout submission
perf(product): optimize filtering query
```

These commits indicate:

- scalability thinking
- production awareness
- backend engineering knowledge
- concurrency understanding

---

## 5. Architecture Commits Matter

AI projects easily become messy.

Architecture commits prove intentional design.

Examples:

```bash
chore(architecture): define layered backend structure
chore(architecture): separate domain and infrastructure layers
```
